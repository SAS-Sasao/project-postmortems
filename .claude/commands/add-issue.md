対象活動: $ARGUMENTS

---

以下の手順で課題を追加せよ。全手順においてCLAUDE.mdのルールを絶対遵守すること。

## 手順

### Step 1: 対象活動の特定とマスタ読み込み

$ARGUMENTS から対象の活動ディレクトリを特定する。

- "djns" または "DJNS" → djns/
- "ai" または "AI推進" → ai-promotion/
- "skill" または "スキル標準化" → skill-standardization/

対象が不明な場合はユーザーに確認を取る。

**特定後、対象ディレクトリの `master.md` を必ず読み込む。**
以降のステップでは master.md に定義された業務種別・影響度・ステータスのみを使用すること。

### Step 2: 既存課題の確認とナンバリング

対象ディレクトリ内の全ステータスディレクトリ（未着手/ 進行中/ 完了/ 保留/）配下の課題mdファイル（NNN_*.md）を走査し、現在の最大番号を取得する。
次の番号を採番する（例: 既存が002なら次は003）。
ファイルが存在しない場合は001から開始する。

**重要: ナンバリングはステータスディレクトリを横断して一意であること。**

### Step 3: ユーザー入力の受付と自動分類

ユーザーから課題内容を受け付ける。入力形式は以下の2パターンに対応する。

#### パターンA: 文章ベースの入力（自動分類）

ユーザーが自由文で課題を記述した場合、master.md の**分類キーワード**列を参照し、以下を自動判定する。

1. **業務種別の自動判定**: 入力文中のキーワードを master.md の分類キーワードと照合し、最も合致度の高い業務種別を選定
2. **影響度の自動判定**: 入力文中のキーワードを master.md の影響度の分類キーワードと照合し、該当するレベルを選定

自動判定の結果をユーザーに提示し、以下を確認する。

```
【自動分類結果】
- 課題タイトル: [入力文から抽出]
- 業務種別: [自動判定結果]（ID: xxx）
- 影響度: [自動判定結果]（ID: x）

この分類で問題ありませんか？修正があれば指定してください。
```

#### パターンB: 項目指定の入力（従来方式）

以下の項目を順番にユーザーへ確認する。選択肢は master.md の定義から提示する。

1. 課題タイトル
2. 業務種別（master.md の業務種別テーブルからID付きで選択肢を提示）
3. 事実（Fact）: 何が起きたか
4. 課題（Problem）: 何が問題だったか
5. 原因（Cause）: なぜ起きたか
6. 影響度（master.md の影響度テーブルからID付きで選択肢を提示）

### Step 4: 課題mdファイルの作成

以下のテンプレートで課題ファイルを作成する。

ファイル名: `NNN_課題タイトル.md`（NNNはゼロ埋め3桁）

**格納先ディレクトリ: ステータスに応じて決定する。**

新規作成時は原則として `未着手/` ディレクトリに格納する。

```
[活動ディレクトリ]/
├── 未着手/       ← 新規課題はここに格納
│   └── NNN_課題タイトル.md
├── 進行中/       ← 対応を開始した課題
├── 完了/         ← 対応完了した課題
└── 保留/         ← 一時保留の課題
```

```markdown
# NNN: [課題タイトル]

- **日付**: YYYY-MM-DD
- **業務種別**: [業務種別名]（ID: [master.mdのID]）
- **事実（Fact）**: [内容]
- **課題（Problem）**: [内容]
- **原因（Cause）**: [内容]
- **影響度**: [レベル]（ID: [master.mdのID]）
- **ステータス**: 未着手（ID: S01）

## 改善アクション

| # | アクション | 期限 | ステータス |
|---|-----------|------|-----------|

## 次の打ち手（仮説）

- TBD
```

### Step 5: README.mdへの反映

1. 対象ディレクトリのREADME.mdの課題テーブルに行を追加し、課題ファイルへのリンクを設定する
   - リンクはステータスディレクトリを含むパスで記載（例: `[001_課題名](未着手/001_課題名.md)`）
2. ルートREADME.mdの該当活動セクションの課題テーブルにも行を追加する

### Step 6: 整合性チェック

リポジトリ全体に対して以下を確認する。

- 課題ファイルのナンバリングに重複がないか（全ステータスディレクトリを横断して確認）
- 課題ファイルが正しいステータスディレクトリに格納されているか（ファイル内のステータスと格納先が一致）
- 全mdファイル内のリンクが正しく解決できるか（ステータスディレクトリを含むパスで確認）
- README.mdのテーブルと実ファイルの整合性が取れているか
- 課題ファイル内の業務種別ID・影響度IDがmaster.mdに存在するか

問題があればユーザーに報告し、修正の確認を取る。

### Step 7: 完了報告

作成した課題の内容を表示し、コミット・プッシュの確認を取る。
