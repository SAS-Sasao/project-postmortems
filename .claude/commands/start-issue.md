対象: $ARGUMENTS

---

以下の手順でGitHub Issueを作成し、課題を「進行中」へ移行せよ。全手順においてCLAUDE.mdのルールを絶対遵守すること。

## 手順

### Step 1: 対象活動の特定

$ARGUMENTS から対象の活動ディレクトリを特定する。

- "djns" または "DJNS" → djns/
- "ai" または "AI推進" → ai-promotion/
- "skill" または "スキル標準化" → skill-standardization/

対象が不明な場合はユーザーに確認を取る。

### Step 2: 未着手課題の探索

$ARGUMENTS から課題を特定する。以下の方法で照合する。

1. **課題番号（ID）指定**: `001` や `003` 等の番号が含まれる場合 → 該当番号の課題ファイルを直接検索
2. **課題名キーワード指定**: キーワードが含まれる場合 → `未着手/` ディレクトリ内の課題ファイル名・内容と照合し、最も合致度の高い課題を候補として提示
3. **曖昧な場合**: `未着手/` ディレクトリ内の全課題一覧を表示し、ユーザーに選択してもらう

**重要: 対象は `未着手/` ディレクトリに格納されている課題のみとする。**

### Step 3: 課題内容のフィードバック確認

特定した課題の内容（タイトル・事実・課題・原因・改善アクション・次の打ち手）を表示し、ユーザーに確認を取る。

```
【対象課題】

- ファイル: NNN_課題タイトル.md
- 業務種別: xxx
- 影響度: x
- 改善アクション: N件
- 次の打ち手: N件

この課題でGitHub Issueを作成しますか？
```

ユーザーの承認後、次のステップへ進む。

### Step 4: GitHub Issue内容の作成

課題ファイルの「改善アクション」と「次の打ち手（仮説）」を基に、**個人の作業レベルまで深掘り・具体化** したIssue内容を作成する。

#### Issue内容の生成ルール

**タイトル**: `[活動名] NNN: 課題タイトル`

**本文の構成**:

```markdown
## 概要

[課題ファイルの事実・課題・原因を簡潔に要約]

## やるべきこと（タスク一覧）

改善アクションと次の打ち手から、個人が実際に手を動かすレベルのタスクに分解する。

- [ ] タスク1: [具体的な作業内容]
- [ ] タスク2: [具体的な作業内容]
- [ ] タスク3: [具体的な作業内容]
...

## 完了条件

- [何をもって完了とするか]

## 参照

- 課題ファイル: [リンク]
```

**タスク分解のルール**:
- 改善アクションの各項目を、1人が1日〜数日で完了できる粒度に分解する
- 「テンプレートを作成する」→「○○の項目を洗い出す」「テンプレートのドラフトを作成する」「レビューを依頼する」等に具体化
- 抽象的な表現は避け、成果物が明確な形にする
- 次の打ち手（仮説）に基づく検証タスクも含める

#### フィードバック確認

作成したIssue内容をユーザーに提示し、確認を取る。

```
【GitHub Issue 内容プレビュー】

タイトル: [活動名] NNN: 課題タイトル

[本文プレビュー]

この内容でGitHub Issueを作成しますか？修正があれば指定してください。
```

ユーザーの承認後、次のステップへ進む。

### Step 5: GitHub Issueの作成

`gh issue create` コマンドを使用してIssueを作成する。

```bash
gh issue create --title "タイトル" --body "本文"
```

作成されたIssueのURLを表示する。

### Step 6: 課題ファイルのステータス更新とディレクトリ移動

1. **課題ファイル内のステータスを更新**: `未着手（ID: S01）` → `進行中（ID: S02）`
2. **ファイルを `未着手/` から `進行中/` へ移動**: `git mv` を使用
3. **README.mdのリンクを更新**: 対象ディレクトリのREADME.mdとルートREADME.mdの課題テーブルのリンクパスとステータスを更新

### Step 7: TODOファイルの作成

Step 4で作成したIssue内容を基に、`進行中/` ディレクトリ内に `NNN_課題タイトル_todo.md` ファイルを作成する。

```markdown
# NNN: [課題タイトル] - TODO

- **GitHub Issue**: [Issue URL]
- **開始日**: YYYY-MM-DD
- **元課題ファイル**: [課題ファイルへのリンク]

## タスク一覧

| # | タスク | 期限 | ステータス |
|---|--------|------|-----------|
| 1 | [具体的な作業内容] | - | 未着手 |
| 2 | [具体的な作業内容] | - | 未着手 |
| 3 | [具体的な作業内容] | - | 未着手 |
...

## 完了条件

- [何をもって完了とするか]

## メモ

- （作業中のメモを記載する場所）
```

### Step 8: 整合性チェック

以下を確認する。

- 課題ファイルが `進行中/` ディレクトリに正しく移動されているか
- 課題ファイル内のステータスが `進行中（ID: S02）` に更新されているか
- README.mdのリンクが `進行中/` パスに更新されているか
- TODOファイルが `進行中/` ディレクトリに作成されているか
- GitHub Issueが正常に作成されているか

問題があればユーザーに報告し、修正の確認を取る。

### Step 9: 完了報告

```
【完了】

- 課題: NNN_課題タイトル.md
- GitHub Issue: [Issue URL]
- TODOファイル: NNN_課題タイトル_todo.md
- ステータス: 未着手 → 進行中
- 格納先: 未着手/ → 進行中/

コミット・プッシュしますか？
```
