# 008: アーキテクチャ・設計思想の未確定のまま開発着手

- **日付**: 2026-03-07
- **業務種別**: 設計（ID: D02）
- **事実（Fact）**: 参画メンバー全員が個別機能の実装に注力し、アプリケーション全体のアーキテクチャ・設計思想・開発ルールを確定しないまま開発フェーズに突入した
- **課題（Problem）**: AI駆動開発で本来必要なアーキテクチャや設計思想が不在のまま開発が進み、各メンバーがバラバラな方針で実装を行った
- **原因（Cause）**: アーキテクチャ・設計思想を策定する専任の担当者を確保せず、全員が機能実装に追われる体制になっていた
- **影響度**: 高（ID: H）
- **ステータス**: 未着手（ID: S01）

## 改善アクション

| # | アクション | 期限 | ステータス |
|---|-----------|------|-----------|
| 1 | プロジェクト体制にアーキテクト役を最低1名確保し、開発着手前にアーキテクチャ・設計思想を確定させることを必須とする | - | 未着手 |
| 2 | 開発着手前に確定すべき設計項目チェックリスト（下記参照）を作成し、全項目が確定するまで開発フェーズに移行しない運用とする | - | 未着手 |
| 3 | 確定した設計思想・開発ルールをドキュメント化し、全メンバーへのオンボーディングを実施する | - | 未着手 |

### 開発着手前に確定すべき設計項目チェックリスト例

| # | 項目 | 内容 |
|---|------|------|
| 1 | アーキテクチャパターン | クリーンアーキテクチャ、レイヤードアーキテクチャ等の選定 |
| 2 | 技術スタック | フレームワーク・言語・ライブラリの確定 |
| 3 | ディレクトリ構成 | プロジェクトのフォルダ構成ルール |
| 4 | 命名規則 | 変数・関数・クラス・ファイル等の命名ルール |
| 5 | エラーハンドリング方針 | 例外処理・エラーレスポンスの統一ルール |
| 6 | API設計方針 | RESTful設計ルール・認証方式 |
| 7 | DB設計方針 | テーブル命名・正規化方針・マイグレーション運用 |

## 次の打ち手（仮説）

- アーキテクト役を明確にアサインすることで、個別機能の実装に全員が没頭する事態を防ぎ、全体の一貫性を保てるのではないか
- 開発着手前チェックリストをゲートとして運用することで、設計が不十分なまま開発に突入するリスクを構造的に排除できるのではないか
