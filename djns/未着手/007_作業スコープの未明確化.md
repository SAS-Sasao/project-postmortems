# 007: 作業スコープの未明確化

- **日付**: 2026-03-07
- **業務種別**: 要件定義（ID: D01）
- **事実（Fact）**: 要件定義までに実装する機能や自社側の作業スコープを明確化できなかった。いつまでに何をすべきか、逆に何をしなくてよいのかが不明確だった
- **課題（Problem）**: 対応優先度が定まらず、チーム全体の作業方針がブレ続けた
- **原因（Cause）**: 要件定義フェーズ開始前にスコープの確定プロセス（やること・やらないことの線引き）が存在しなかった
- **影響度**: 高（ID: H）
- **ステータス**: 未着手（ID: S01）

## 改善アクション

| # | アクション | 期限 | ステータス |
|---|-----------|------|-----------|
| 1 | 要件定義開始前に「スコープ定義書」を作成し、対応範囲と対応外範囲を明文化するプロセスを導入する（下記参照） | - | 未着手 |
| 2 | スコープ定義書はステークホルダーの承認を得た上で、以降の変更はスコープ変更管理プロセスを経る運用とする | - | 未着手 |
| 3 | スコープ定義書をもとに優先度付き機能一覧（MoSCoW分析等）を作成し、チーム全員に共有する | - | 未着手 |

### スコープ定義書の記載項目例

| # | 項目 | 内容 |
|---|------|------|
| 1 | プロジェクト目的 | 本案件で達成すべきビジネス目標 |
| 2 | 対応範囲（In Scope） | 実装する機能・対応する業務領域の一覧 |
| 3 | 対応外範囲（Out of Scope） | 今回対応しない機能・業務の一覧と理由 |
| 4 | 前提条件 | スコープの前提となる条件 |
| 5 | 制約事項 | 予算・期間・技術的制約 |
| 6 | 優先度 | Must / Should / Could / Won't（MoSCoW） |

## 次の打ち手（仮説）

- 「やらないこと」を明示的に定義・合意することで、スコープクリープを防ぎ、チームが迷わず優先度の高い作業に集中できるのではないか
- MoSCoW分析で機能の優先度を可視化することで、リソース不足時にも「何を削るか」の判断を迅速に行えるのではないか
