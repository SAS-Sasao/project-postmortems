# 009: WBS管理の形骸化

- **日付**: 2026-03-07
- **業務種別**: PM・管理（ID: D08）
- **事実（Fact）**: PMOが参画するまで、WBSはただ作るだけで実態とかけ離れた内容になっていた。各メンバーは進捗を更新できず、プロジェクトの状況が可視化されない状態が続いた
- **課題（Problem）**: WBSが形骸化し進捗管理の機能を果たさなかったため、遅延の検知や対策が後手に回った
- **原因（Cause）**: WBSの粒度が実態と合わず更新しづらい状態だった上、進捗更新のルール・頻度・責任者が定められていなかった
- **影響度**: 高（ID: H）
- **ステータス**: 未着手（ID: S01）

## 改善アクション

| # | アクション | 期限 | ステータス |
|---|-----------|------|-----------|
| 1 | WBSのタスク粒度を「1タスク = 1〜3日で完了できる」レベルに統一し、各メンバーが更新しやすい粒度に分解する | - | 未着手 |
| 2 | 週次で進捗更新を行う定例ミーティングを設け、WBS更新を必須アジェンダとする | - | 未着手 |
| 3 | WBSの各タスクに担当者・開始日・終了予定日を必ず設定し、未更新が3日以上続いた場合はPMがフォローする運用とする | - | 未着手 |

## 次の打ち手（仮説）

- タスク粒度を適切に分解し更新頻度を定めることで、WBSと実態の乖離を防ぎ、メンバーが「更新できない」という状態を解消できるのではないか
- 未更新アラートの仕組みを設けることで、進捗の停滞を早期に検知し、遅延が深刻化する前に対策を打てるのではないか
