# 004: インナーソース管理体制の未確立

- **日付**: 2026-03-07
- **業務種別**: 開発（ID: K04）
- **事実（Fact）**: 受託案件で開発したアプリケーションや1次請けで参画した案件の情報が社内でシェアされておらず、開発資産を活用しきれていない
- **課題（Problem）**: 過去に開発したソースコードやドキュメントが散在しており、社内の他メンバーがアクセス・再利用できない状態にある
- **原因（Cause）**: ソースコードの一元管理（GitHub等）やドキュメントの共有基盤が確立されておらず、案件終了後に資産が埋もれる構造になっていた
- **影響度**: 高（ID: H）
- **ステータス**: 未着手（ID: S01）

## 改善アクション

| # | アクション | 期限 | ステータス |
|---|-----------|------|-----------|
| 1 | 社内GitHubオーガニゼーションを整備し、受託案件のソースコードを管理するルール・リポジトリ構成を策定する | - | 未着手 |
| 2 | 案件関連ドキュメント（設計書・手順書等）をいつでも社内メンバーが閲覧できる管理基盤（SharePoint・Confluence等）を構築する | - | 未着手 |
| 3 | 案件完了時にソースコード・ドキュメントを所定の場所に格納するプロセスを案件クローズ手順に組み込む | - | 未着手 |

## 次の打ち手（仮説）

- インナーソース管理を確立することで、新規案件で過去の実装を参考にでき、開発効率とコード品質の向上に繋がるのではないか
- 将来的にデータレイク・データウェアハウスで案件情報を統合管理することで、営業レポートやマーケティング活動に活用できるデータドリブンな経営戦略への移行が可能になるのではないか
