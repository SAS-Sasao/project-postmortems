# 007: 過去案件の失敗事例・課題の未整理

- **日付**: 2026-03-08
- **業務種別**: 方法論策定（ID: K08）
- **事実（Fact）**: 過去の案件やプロジェクトにおける失敗事例や課題がまとまっておらず、成功例だけでなく失敗から学ぶ仕組みが存在しない
- **課題（Problem）**: 過去の失敗や課題が共有されないため、同じ失敗が繰り返されるリスクがあり、組織としての学習が進んでいない
- **原因（Cause）**: 失敗事例を記録・共有するためのフォーマット・管理基盤・運用ルールが整備されておらず、かつ失敗を共有することへの心理的安全性が担保されていなかった
- **影響度**: 高（ID: H）
- **ステータス**: 未着手（ID: S01）

## 改善アクション

| # | アクション | 期限 | ステータス |
|---|-----------|------|-----------|
| 1 | 失敗事例・課題の記録テンプレート（下記参照）を作成し、案件完了時の事後振り返りを必須プロセスとして組み込む | - | 未着手 |
| 2 | 失敗事例を一元管理する基盤（SharePoint等の専用セクション）を構築する | - | 未着手 |
| 3 | 「失敗を記載した人の評価を下げない。むしろ感謝し評価する」という方針を明文化し、社内に周知する | - | 未着手 |
| 4 | 四半期ごとに失敗事例の振り返り共有会を実施し、組織全体で学びを共有する場を設ける | - | 未着手 |

### 失敗事例記録テンプレート例

| # | 項目 | 内容 |
|---|------|------|
| 1 | 案件名 | 対象の案件・プロジェクト名 |
| 2 | 案件全体の評価 | 成功 / 一部課題あり / 失敗 |
| 3 | 失敗・課題の概要 | 何が問題だったか |
| 4 | 原因分析 | なぜ起きたか |
| 5 | 影響範囲 | どの程度の影響があったか |
| 6 | 得られた教訓 | 次に活かすべきこと |
| 7 | 推奨アクション | 今後のプロジェクトでの具体的対策 |

## 次の打ち手（仮説）

- 事後振り返りを必須プロセス化し心理的安全性を明文化することで、失敗を隠さず共有する文化が醸成され、組織全体の学習速度が向上するのではないか
- 失敗事例を一元管理し定期的な共有会を行うことで、異なるプロジェクトチーム間でも教訓が伝播し、同じ失敗の再発を構造的に防げるのではないか
